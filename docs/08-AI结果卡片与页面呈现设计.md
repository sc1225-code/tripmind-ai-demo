# TripMind AI 结果卡片与页面呈现设计

## 1. 文档目的

本文档用于说明 TripMind AI 的 AI 输出结果如何在移动端页面中呈现。

上一阶段已经完成：

- AI 能力设计
- AI 输出样例设计
- 行程规划、知识库、Plan B 的核心内容结构

本阶段重点解决：

- 长 AI 回复如何拆成移动端可读的卡片。
- 哪些信息默认展示，哪些信息折叠。
- 用户可以对 AI 结果执行哪些操作。
- 如何在页面中轻量展示来源、置信度、不确定项和确认清单。

设计原则：

```text
先给结论
再给对比
重要依据可见
详细来源可折叠
高风险操作必须确认
```

## 2. AI 结果页面总体结构

AI 结果仍然出现在 AI 问答主入口中，不新增独立结果页。

用户发送问题后，聊天区域从欢迎态切换为结果态：

```text
┌────────────────────────────────┐
│ ☰          TripMind AI     小林 │
├────────────────────────────────┤
│ 用户：新疆和云南怎么选？        │
│                                │
│ AI 结果卡片组                  │
│ ┌────────────────────────────┐ │
│ │ 推荐结论卡                  │ │
│ └────────────────────────────┘ │
│ ┌────────────────────────────┐ │
│ │ 多维对比卡                  │ │
│ └────────────────────────────┘ │
│ ┌────────────────────────────┐ │
│ │ 为什么适合你                │ │
│ └────────────────────────────┘ │
│ ┌────────────────────────────┐ │
│ │ 备选方案                    │ │
│ └────────────────────────────┘ │
│ ┌────────────────────────────┐ │
│ │ 来源 / 不确定项             │ │
│ └────────────────────────────┘ │
├────────────────────────────────┤
│ [行程规划*] [Plan B] [知识库]   │
│ ┌────────────────────────────┐ │
│ │ 语音  继续追问或调整... + 发送 │ │
│ └────────────────────────────┘ │
└────────────────────────────────┘
```

页面规则：

- AI 问答仍是主入口，不跳转到单独报告页。
- 结果以“卡片组”呈现，而不是一整段长文本。
- 输入框常驻底部，方便用户继续追问或调整。
- 模式按钮保留当前激活态，例如“行程规划*”。
- 结果卡片可以上下滚动，底部输入区固定。

## 3. 卡片类型总览

| 卡片类型 | 使用场景 | 默认状态 | 作用 |
| --- | --- | --- | --- |
| 意图识别提示条 | 用户发送问题后 | 展开 | 告诉用户 AI 识别到的模式 |
| 推荐结论卡 | 行程规划 | 展开 | 先给核心建议和置信度 |
| 多维对比卡 | 行程规划 | 展开 | 通过横向滑动维度卡展示新疆 vs 云南关键对比 |
| 对比结论 | 行程规划 | 展开 | 放在多维对比卡底部，解释为什么更适合用户本人 |
| 备选方案卡 | 行程规划、Plan B | 展开 | 给用户保留选择空间 |
| 来源卡 | 所有重要 AI 输出 | 折叠 | 展示引用资料、记忆、工具数据 |
| 不确定项卡 | 行程规划、Plan B | 折叠 | 告诉用户哪些信息仍需确认 |
| 下一步操作卡 | 行程规划 | 展开 | 提供继续生成或调整入口 |
| 影响判断卡 | Plan B | 展开 | 说明突发事件影响范围 |
| 执行确认清单卡 | Plan B | 展开 | 承接高风险操作前确认 |
| 边界说明卡 | Plan B、高风险操作 | 折叠 | 说明 AI 不会自动执行什么 |

## 4. 行程规划结果卡片设计

### 4.1 结果卡片顺序

行程规划结果建议按以下顺序展示：

```text
意图识别提示条
推荐结论卡
多维对比卡
备选方案卡
下一步操作卡
来源卡
不确定项卡
```

排序逻辑：

- 用户先看到结论，降低阅读成本。
- 对比表紧跟结论，支撑用户判断。
- 对比结论让用户理解“为什么是我适合这个”。
- 来源和不确定项保留可信信息，但默认不抢主视线。

### 4.2 意图识别提示条

```text
┌────────────────────────────┐
│ 已按「行程规划」分析        │
│ 参考你的资料、记忆和模拟天气 │
└────────────────────────────┘
```

规则：

- 用轻量提示条，不做大卡片。
- 如果用户手动点亮模式，文案为“已按你选择的「行程规划」分析”。
- 如果系统自动识别模式，文案为“已自动识别为「行程规划」问题”。
- 用户可以点提示条中的模式名，查看为什么进入该模式。

### 4.3 推荐结论卡

```text
┌────────────────────────────┐
│ 推荐结论                    │
│ 更推荐：云南 15 天游学       │
│                              │
│ 更符合预算、学习时间和交通强度 │
│ 要求；新疆自驾体验更强，但对   │
│ 体力和天气稳定性要求更高。     │
│                              │
│ 置信度：较高                 │
└────────────────────────────┘
```

内容规则：

- 标题固定为“推荐结论”。
- 结论必须在第一屏可见。
- 置信度采用“较高 / 中等 / 需要确认”，不使用复杂百分比。
- 不在这里放太多解释，详细原因放到多维对比卡底部的对比结论中。

### 4.4 多维对比卡

移动端不适合完整横向大表格，也不优先采用雷达图。雷达图适合展示多维轮廓，但在手机上容易出现面积误读和读数困难。

当前建议采用“横向滑动维度卡片 + 双方案评分条 + 胜出标签”的形式。参考 ChatGPT 移动端“最新报道”一类横向内容模块，优点是减少纵向滚动压力，同时保留多维信息的完整性。

```text
┌────────────────────────────┐
│ 多维对比                    │
│ 横向滑动查看 7 个决策维度。  │
│                              │
│ ┌──────────────┐ ┌────────── │
│ │预算成本  云南 │ │时间节奏   │
│ │新疆 ███ 62   │ │新疆 ███   │
│ │云南 ████ 86  │ │云南 ████  │
│ │云南更易控预算│ │云南更稳... │
│ └──────────────┘ └────────── │
│                              │
│ 结论：云南更适合你这次出行。  │
└────────────────────────────┘
```

默认可见：

- 第 1 张完整维度卡
- 第 2 张维度卡的一部分，用于提示可横向滑动

横向滑动后依次展示：

- 预算成本
- 时间节奏
- 交通强度
- 天气季节
- 安全风险
- 体验价值
- 学习适配

规则：

- 每张维度卡只承载一个决策维度。
- 每张维度卡控制在固定高度内，避免继续拉长页面。
- 不做横向滑动表格，避免移动端阅读负担。
- 不优先做雷达图，避免手机端读数困难。
- “更适合”用短标签展示，例如“云南”“新疆”“看偏好”。

### 4.4.1 对比结论

“为什么适合你”不再单独作为独立卡片，而是放在多维对比卡底部，作为对比后的结论展示。

```text
┌────────────────────────────┐
│ 结论：云南更适合你这次出行。  │
│ 它在预算、节奏、交通强度和学习 │
│ 适配上更稳定；新疆自然体验更强，│
│ 但更适合作为低强度备选路线。  │
└────────────────────────────┘
```

规则：

- 只展示与用户本人相关的理由。
- 避免写成通用攻略。
- 结论跟随对比卡出现，减少用户在多个卡片之间来回理解。

### 4.5 备选方案卡

```text
┌────────────────────────────┐
│ 备选方案                    │
│                              │
│ A 云南慢节奏游学             │
│ 昆明 3 天 → 大理 5 天 →      │
│ 丽江 4 天 → 返程缓冲 3 天    │
│ [生成详细日程]               │
│                              │
│ B 新疆低强度版本             │
│ 不做全程自驾，只保留 2-3 个   │
│ 核心目的地。                 │
│ [查看备选路线]               │
└────────────────────────────┘
```

规则：

- 备选方案至少 2 个。
- 每个方案有一个明确按钮。
- 按钮不直接执行高风险操作，只生成进一步方案。

### 4.6 下一步操作卡

```text
┌────────────────────────────┐
│ 下一步                      │
│ [生成云南 15 天游学日程]     │
│ [生成新疆低强度备选路线]     │
│ [调整预算]                  │
│ [补充出发城市]              │
└────────────────────────────┘
```

规则：

- 按钮用于继续对话，不跳出 AI 问答。
- 点击按钮后，将对应意图填入输入框或直接发送。
- 不超过 4 个操作，避免选择过多。

### 4.7 来源与不确定项卡

```text
┌────────────────────────────┐
│ 来源与不确定项              │
│ 参考 4 份资料、4 条记忆、3 个 │
│ 模拟工具数据                │
│                              │
│ [查看来源] [查看不确定项]     │
└────────────────────────────┘
```

展开来源：

```text
资料源
- 云南游学攻略.pdf
- 新疆 15 天自驾笔记.jpg
- 个人预算计划.xlsx
- 课程安排截图.png

AI 记忆
- 总预算不超过 15000 元
- 每天保留 2 小时学习时间
- 容易晕车
- 偏好慢节奏

模拟工具数据
- 目的地天气趋势
- 城市间交通时间
- 住宿价格区间
```

展开不确定项：

```text
- 出发城市
- 具体出行月份
- 是否有同行人
- 实时机票和住宿价格
- 新疆部分路线开放状态
```

规则：

- 来源和不确定项默认折叠。
- 卡片摘要必须让用户知道 AI 不是凭空生成。
- 来源分为资料源、AI 记忆、工具数据、模型推理。

## 5. Plan B 结果卡片设计

### 5.1 结果卡片顺序

Plan B 结果建议按以下顺序展示：

```text
意图识别提示条
影响判断卡
替代方案卡
成本与时间变化卡
执行确认清单卡
来源卡
边界说明卡
```

排序逻辑：

- 用户遇到突发时最先关心影响有多大。
- 替代方案紧跟影响判断。
- 执行确认清单承接真实操作前的用户确认。

### 5.2 影响判断卡

```text
┌────────────────────────────┐
│ 影响判断                    │
│ 明天大理降雨会影响环洱海骑行 │
│                              │
│ 影响范围：Day 5 下午         │
│ 风险：雨天骑行安全风险升高    │
│ 建议：不要执行完整骑行        │
│                              │
│ 置信度：中等偏高             │
└────────────────────────────┘
```

规则：

- 明确“影响什么”和“不建议什么”。
- 不把替代方案放在这张卡里，避免信息过载。
- 置信度必须可见。

### 5.3 替代方案卡

```text
┌────────────────────────────┐
│ 替代方案                    │
│                              │
│ A 低风险学习日               │
│ 咖啡馆学习 + 室内展馆 + 笔记整理 │
│ 适合：保证学习进度            │
│ [采用]                       │
│                              │
│ B 轻量体验日                 │
│ 古城短距离步行 + 室内体验      │
│ 适合：保留一点旅行体验        │
│ [采用]                       │
│                              │
│ C 顺延骑行                   │
│ 明天室内安排，后天再骑行       │
│ 适合：环洱海优先级很高        │
│ [查看可行性]                 │
└────────────────────────────┘
```

规则：

- 至少提供 2 个方案，最多 3 个。
- 每个方案都说明“适合谁”。
- “采用”不代表真实执行订单，只进入确认清单。

### 5.4 成本与时间变化卡

```text
┌────────────────────────────┐
│ 成本与时间变化              │
│ 新增：打车 50-120 元         │
│ 新增：室内体验 50-200 元      │
│ 节省：骑行租车费用            │
│ 时间：可保留 2-3 小时学习     │
└────────────────────────────┘
```

规则：

- 使用范围估算，不假装精确。
- 如果数据来自模拟工具，需要在来源卡中说明。
- 成本变化和时间变化分开写。

### 5.5 执行确认清单卡

```text
┌────────────────────────────┐
│ 执行前确认                  │
│ □ 骑行预约是否可取消或改期    │
│ □ 后天天气是否适合顺延        │
│ □ 明天是否有必须完成的学习任务 │
│ □ 是否接受新增交通费用        │
│                              │
│ [生成调整后日程]              │
└────────────────────────────┘
```

规则：

- 每一项是用户需要确认的真实事项。
- 勾选只是记录用户确认，不代表 AI 自动操作。
- 按钮文案必须避免“立即改签”“自动取消”等高风险表达。

### 5.6 边界说明卡

```text
┌────────────────────────────┐
│ AI 边界说明                 │
│ 我可以帮你生成调整建议和清单， │
│ 但不会自动取消预约、改签订单或 │
│ 通知同行人。                 │
└────────────────────────────┘
```

规则：

- 默认折叠在来源卡下方。
- 当用户点击“采用方案”或涉及订单调整时，可以临时展开。

## 6. 知识库结果卡片设计

知识库相关 AI 输出主要出现在两个位置：

- 用户上传资料后。
- 用户点亮“知识库”模式提问后。

### 6.1 资料识别结果卡

```text
┌────────────────────────────┐
│ 已整理 4 份资料             │
│                              │
│ PDF 云南游学攻略.pdf         │
│ 攻略 · 提取 12 条知识         │
│ 可用于：路线规划、学习空间    │
│                              │
│ 图 新疆 15 天自驾笔记.jpg     │
│ 攻略截图 · 提取 9 条知识      │
│ 可用于：交通强度、路线判断    │
│                              │
│ [查看全部资料源]              │
└────────────────────────────┘
```

规则：

- 默认只展示前 2 个资料源。
- 资料源点击后进入知识库详情，或在当前卡片展开摘要。
- “提取知识”只显示数量，不在主卡片中铺开全部内容。

### 6.2 AI 记忆确认卡

```text
┌────────────────────────────┐
│ 发现可保存的 AI 记忆         │
│                              │
│ 预算与时间                   │
│ □ 总预算不超过 15000 元       │
│ □ 每天保留 2 小时学习时间     │
│                              │
│ 饮食与身体限制               │
│ □ 不吃牛羊肉                 │
│ □ 容易晕车                   │
│                              │
│ [全部保存] [逐条确认]         │
└────────────────────────────┘
```

规则：

- 长期记忆必须经过用户确认。
- 默认勾选可以讨论，但 MVP 建议默认不勾选，强调用户控制感。
- 每条记忆后续支持来源追溯。

### 6.3 知识库问答引用卡

```text
┌────────────────────────────┐
│ 基于知识库回答              │
│ 根据你上传的云南攻略和预算表，│
│ 大理更适合安排固定学习窗口。  │
│                              │
│ 引用来源：2 份资料、2 条记忆   │
│ [查看引用]                   │
└────────────────────────────┘
```

规则：

- 知识库模式下必须显示引用提示。
- “查看引用”展开后展示具体资料名和关联片段摘要。
- 如果知识库没有相关资料，AI 应明确提示“未找到可引用资料”。

## 7. 卡片状态与交互规则

### 7.1 默认展开与折叠

默认展开：

- 推荐结论
- 多维对比横向卡片组
- 对比结论
- 备选方案
- 影响判断
- 替代方案
- 执行确认清单

默认折叠：

- 完整来源
- 不确定项
- 边界说明
- 全量资料源
- 全量记忆来源

### 7.2 按钮行为

按钮分为三类：

| 类型 | 示例 | 行为 |
| --- | --- | --- |
| 继续生成 | 生成云南 15 天游学日程 | 触发下一轮 AI 输出 |
| 补充信息 | 补充出发城市 | 将问题填入输入框或打开轻量输入 |
| 确认类 | 保存记忆、生成调整后日程 | 先确认用户选择，再继续生成 |

禁止在 MVP 中出现的按钮：

- 立即付款
- 自动改签
- 一键退票
- 自动通知同行人
- 自动下单

### 7.3 置信度展示

置信度不使用复杂评分，采用轻量标签：

```text
较高：资料和约束充分，结论较稳定
中等：有依据，但仍受实时信息影响
需要确认：缺少关键输入或实时数据
```

### 7.4 来源展示

来源分为四类：

```text
资料源：用户上传的文件、截图、表格
AI 记忆：用户确认保存的长期偏好和约束
工具数据：天气、地图、价格、景点状态等
模型推理：基于以上信息做出的综合判断
```

来源展示不需要占据主视线，但必须可查看。

## 8. Demo 页面落地建议

当前 `demo/mobile-v2/` 可以逐步增加以下状态：

1. AI 问答发送后出现行程规划结果卡片组。
2. 横向滑动查看完整多维对比。
3. 点击“查看来源”展开来源卡。
4. 点击“Plan B”模式并发送天气问题后，展示 Plan B 卡片组。
5. 在知识库页上传资料后，展示资料识别结果和 AI 记忆确认卡。
