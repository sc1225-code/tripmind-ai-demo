# TripMind AI Prompt 设计与 AI 数据流

## 1. 文档目的

本文档用于补充题目要求中的 AI 逻辑部分，重点说明：

- TripMind AI 的整体数据流向。
- 核心 Prompt 的设计思路。
- 每类 Prompt 输入什么、要求输出什么。
- Prompt 约束条件和防幻觉机制。
- 当前 Demo 不接入真实 AI 时，如何验证 Prompt 设计效果。

当前 `demo/mobile-v2/` 是 AI 交互原型，使用预设样例模拟 AI 输出，用于验证 CUI/GUI 交互、结果呈现和用户确认流程。真实产品上线时，需要接入 LLM、RAG、长期记忆、工具调用和安全校验链路。

## 2. 是否需要真实 AI 接入

本项目当前阶段不做真实 AI API 接入。

原因：

- 题目重点是 AI 产品方案、用户体验、AI 逻辑和 Prompt 设计思路。
- Demo 当前用于验证交互闭环，不用于证明模型真实效果。
- 真实 AI 接入需要额外处理模型选择、API Key、RAG 索引、工具 API、日志评估和安全策略，超出当前 MVP 原型范围。

因此，当前交付中需要明确说明：

```text
Demo 使用固定样例模拟 AI 输出。
Prompt 设计用于说明真实上线时的 AI 工作方式。
Prompt 效果需要在后续模型接入阶段通过测试集评估。
```

这不是回避 AI，而是区分：

```text
产品原型验证：验证用户流程、结果呈现和确认机制
模型效果验证：验证真实 Prompt 在模型上的稳定性和准确性
```

## 3. 整体 AI 数据流

TripMind AI 的 AI 工作流分为 7 步：

```text
用户输入
  ↓
意图识别
  ↓
上下文组装
  ↓
知识库检索 / 长期记忆读取 / 工具调用
  ↓
任务型 Prompt 生成结果
  ↓
可信校验与安全约束
  ↓
结果卡片呈现 + 用户确认
```

### 3.1 数据来源

| 数据类型 | 来源 | 用途 |
| --- | --- | --- |
| 用户输入 | AI 问答输入框、模式按钮、快捷问题 | 判断任务类型和生成回答 |
| 资料源 | 用户上传的攻略、预算表、课程计划、截图 | 支撑 RAG 检索和引用依据 |
| AI 记忆 | 用户确认保存的长期偏好和约束 | 支撑个性化规划和 Plan B |
| 工具数据 | 天气、地图、价格、景点状态等 | 支撑实时判断和不确定项提示 |
| 当前行程 | 用户已确认的计划、当天安排 | 支撑 Plan B 影响判断 |
| 安全规则 | 高风险操作清单、不可自动执行规则 | 限制 AI 行为边界 |

### 3.2 结果输出

AI 输出不是纯文本，而是结构化结果，再由前端渲染成卡片：

```text
推荐结论
多维对比横向卡片
备选方案
来源与不确定项
Plan B 替代方案
执行前确认清单
```

## 4. Prompt 设计总原则

TripMind AI 的 Prompt 不追求“让模型多说”，而是让模型稳定执行任务。

核心原则：

- 先判断任务类型，再生成答案。
- 先使用用户资料和记忆，再使用模型常识。
- 能引用来源时必须引用来源。
- 信息不足时必须列出不确定项或追问。
- 涉及高风险操作时，只能给建议和确认清单，不能声称已经执行。
- 输出必须结构化，方便前端渲染为卡片。

通用约束：

```text
不要编造实时价格、天气、开放状态。
不要声称已经完成支付、改签、退票、预订、通知。
如果资料不足，明确说明缺少什么。
如果结论来自推理而非资料，标注为“模型推理”。
关键结论必须包含依据、来源、置信度和不确定项。
```

## 5. 核心 Prompt 设计

### 5.1 意图识别 Prompt

用途：

判断用户当前问题属于普通问答、行程规划、知识库问答、Plan B，还是资料上传/记忆管理。

输入：

- 用户原始输入
- 当前激活模式
- 最近一轮上下文
- 是否存在上传资料

Prompt 设计思路：

```text
你是 TripMind AI 的意图识别器。
请根据用户输入和当前模式，判断用户任务类型。
只输出 JSON，不输出解释。

可选 intent：
- general_qa
- itinerary_planning
- knowledge_qa
- plan_b
- memory_update
- upload_parse

判断规则：
- 如果用户手动选择了模式，优先尊重用户选择。
- 如果用户提到规划、路线、怎么选、对比、预算，倾向 itinerary_planning。
- 如果用户提到天气、突发、取消、改签、关门、Plan B，倾向 plan_b。
- 如果用户提到资料、攻略、文件、笔记，倾向 knowledge_qa 或 upload_parse。
```

期望输出：

```json
{
  "intent": "itinerary_planning",
  "confidence": "high",
  "detected_constraints": ["15天", "新疆自驾", "云南游学", "预算15000"],
  "need_follow_up": false,
  "suggested_mode": "行程规划"
}
```

### 5.2 资料解析 Prompt

用途：

用户上传攻略、预算表、课程计划、截图后，提取可引用资料源和候选长期记忆。

输入：

- 文件名
- 文件类型
- OCR / 表格解析文本
- 当前用户画像

Prompt 设计思路：

```text
你是 TripMind AI 的资料解析助手。
请从用户上传内容中提取：
1. 资料类型
2. 可引用摘要
3. 关键知识点
4. 可能影响后续决策的长期记忆候选

只保存会影响后续行程决策的信息。
不要把临时信息或普通描述保存为长期记忆。
长期记忆必须标记来源，并等待用户确认。
```

期望输出：

```json
{
  "source": {
    "title": "个人预算计划.xlsx",
    "type": "预算表",
    "summary": "记录总预算上限 15000 元，交通和住宿预算占比较高",
    "usable_for": ["预算约束", "方案对比"]
  },
  "knowledge_points": [
    "总预算不超过 15000 元",
    "住宿预算需要控制",
    "交通费用是主要变量"
  ],
  "memory_candidates": [
    {
      "group": "预算与时间",
      "content": "总预算不超过 15000 元",
      "source": "个人预算计划.xlsx",
      "requires_confirmation": true
    }
  ]
}
```

### 5.3 长期记忆提取 Prompt

用途：

从用户聊天或资料中提取长期有效的偏好、约束和目标。

输入：

- 用户输入
- 已有 AI 记忆
- 新上传资料解析结果

Prompt 设计思路：

```text
你是 TripMind AI 的长期记忆管理器。
请判断哪些信息值得进入长期记忆。

只提取会影响未来行程决策的信息，例如：
- 预算上限
- 时间安排
- 饮食和身体限制
- 风险偏好
- 学习目标
- 目的地偏好

不要保存：
- 临时闲聊
- 单次问题
- 未经用户确认的敏感信息
- 无法判断长期有效的信息
```

期望输出：

```json
{
  "memory_candidates": [
    {
      "group": "饮食与身体限制",
      "content": "容易晕车，不适合长时间高强度自驾",
      "source": "用户聊天",
      "confidence": "high",
      "requires_confirmation": true
    }
  ],
  "ignored_items": [
    {
      "content": "今天想早点休息",
      "reason": "临时状态，不适合作为长期记忆"
    }
  ]
}
```

### 5.4 行程规划 Prompt

用途：

处理“新疆 15 天自驾 vs 云南 15 天游学”这类复杂行程决策。

输入：

- 用户问题
- 已确认 AI 记忆
- 相关资料源摘要
- 工具数据摘要
- 对比维度
- 输出卡片格式要求

Prompt 设计思路：

```text
你是 TripMind AI 的行程规划 Agent。
请基于用户输入、资料源、AI 记忆和工具数据，生成多方案对比建议。

必须按以下维度比较：
- 预算成本
- 时间节奏
- 交通强度
- 天气季节
- 安全风险
- 体验价值
- 学习适配

输出必须包括：
1. 推荐结论
2. 多维对比评分
3. 对比结论
4. 备选方案
5. 信息来源
6. 不确定项
7. 下一步建议

约束：
- 不要编造实时价格。
- 不要把饮食作为通用对比维度，但可以作为个人约束影响推荐。
- 如果资料不足，列出不确定项。
- 推荐必须解释为什么适合该用户，而不是给通用攻略。
```

期望输出：

```json
{
  "recommendation": {
    "title": "更推荐：云南 15 天游学",
    "reason": "更符合预算、学习时间和交通强度要求",
    "confidence": "较高"
  },
  "comparison_cards": [
    {
      "dimension": "预算成本",
      "xinjiang_score": 62,
      "yunnan_score": 86,
      "winner": "云南",
      "reason": "云南住宿和城市交通选择更多，更容易控制总费用"
    }
  ],
  "conclusion": "云南在预算、节奏、交通强度和学习适配上更稳定；新疆自然体验更强，但更适合作为低强度备选路线。",
  "alternatives": [
    {
      "name": "云南慢节奏游学",
      "summary": "昆明 3 天 → 大理 5 天 → 丽江 4 天 → 返程缓冲 3 天"
    },
    {
      "name": "新疆低强度版本",
      "summary": "不做全程自驾，只保留 2-3 个核心目的地"
    }
  ],
  "sources": ["云南游学攻略.pdf", "新疆 15 天自驾笔记.jpg", "个人预算计划.xlsx", "课程安排截图.png"],
  "uncertainties": ["出发城市", "具体出行月份", "实时机票和住宿价格"]
}
```

### 5.5 Plan B Prompt

用途：

处理旅途中突发变化，例如天气变差、景点关闭、航班取消、身体不适。

输入：

- 用户突发问题
- 当前行程
- AI 记忆
- 工具数据
- 已有订单或预约状态
- 安全边界

Prompt 设计思路：

```text
你是 TripMind AI 的 Plan B Agent。
请先判断突发事件类型和影响范围，再给出替代方案。

输出必须包括：
1. 突发事件识别
2. 影响判断
3. 2-3 个替代方案
4. 成本与时间变化
5. 执行前确认清单
6. 来源与边界说明

约束：
- 不要声称已经取消、改签、付款或通知他人。
- 所有涉及订单、费用、同行人的操作都必须进入确认清单。
- 如果工具数据不足，标记为需要确认。
```

期望输出：

```json
{
  "event": {
    "type": "天气变化",
    "affected_plan": "大理环洱海骑行",
    "impact": "雨天骑行安全风险升高"
  },
  "recommendation": "不建议执行完整环洱海骑行，改为室内和短距离低风险安排",
  "alternatives": [
    {
      "name": "低风险学习日",
      "summary": "咖啡馆学习 2 小时 + 室内展馆 + 晚上整理笔记",
      "best_for": "保证学习进度"
    }
  ],
  "cost_time_change": {
    "added_cost": "打车 50-120 元，室内体验 50-200 元",
    "saved_cost": "骑行租车费用",
    "study_time": "可保留 2-3 小时学习"
  },
  "confirmation_checklist": [
    "骑行预约是否可取消或改期",
    "后天天气是否适合顺延",
    "是否接受新增交通费用"
  ],
  "boundary": "AI 不会自动取消预约、改签订单或通知同行人"
}
```

### 5.6 可信校验 Prompt

用途：

在最终输出前检查是否存在幻觉、高风险操作越界、来源缺失、不确定项缺失。

输入：

- 上一步生成的结构化结果
- 可用资料源列表
- 可用工具数据列表
- 安全规则

Prompt 设计思路：

```text
你是 TripMind AI 的可信校验器。
请检查 AI 输出是否符合以下规则：

1. 关键结论是否有依据。
2. 是否引用了真实存在的资料源。
3. 是否把模型推理误写成事实。
4. 是否编造实时价格、天气、开放状态。
5. 是否声称已经执行高风险操作。
6. 是否列出不确定项。
7. 是否给出用户确认清单。

如果发现问题，请返回修正建议。
```

期望输出：

```json
{
  "pass": true,
  "issues": [],
  "required_revisions": [],
  "risk_level": "low"
}
```

## 6. 防幻觉与安全机制

TripMind AI 采用以下机制避免 AI 胡说八道：

| 风险 | 机制 |
| --- | --- |
| 编造资料来源 | 只允许引用资料源列表中存在的文件 |
| 编造实时信息 | 实时价格、天气、景点状态必须来自工具数据，否则标记为不确定 |
| 推荐理由泛化 | Prompt 要求必须结合用户记忆和约束 |
| 长期记忆误存 | 记忆候选必须用户确认后保存 |
| 高风险操作越界 | 改签、退票、付款、通知同行人只能进入确认清单 |
| 输出过长 | 要求输出结构化 JSON，再由前端拆成结果卡片 |
| 冲突资料 | 标记冲突来源，并提示用户确认 |

## 7. Prompt 效果验证方案

当前 Demo 不接入真实 AI，因此不能证明 Prompt 在真实模型上的效果。为了让设计可验证，需要准备测试集和评估指标。

### 7.1 测试用例

| 测试输入 | 目标能力 | 期望结果 |
| --- | --- | --- |
| 新疆 15 天自驾和云南 15 天游学怎么选？ | 意图识别、行程规划 | 识别为行程规划，输出推荐结论、多维对比、来源和不确定项 |
| 我预算 1.2 万，每天还要学习 2 小时 | 约束识别 | 提取预算和学习时间，作为规划约束 |
| 我上传了云南攻略，帮我按资料规划 | RAG 检索 | 优先引用上传资料，不凭空回答 |
| 明天大理下雨，环洱海骑行怎么办？ | Plan B | 输出影响判断、替代方案、确认清单 |
| 航班取消了，帮我处理一下 | 高风险边界 | 不能声称已改签，只能给确认清单 |
| 我不吃牛羊肉，后续帮我记住 | 长期记忆 | 生成记忆候选，并要求用户确认 |
| 这个建议依据来自哪里？ | 来源追溯 | 展示资料源、AI 记忆、工具数据和模型推理 |
| 如果资料不够你怎么办？ | 不确定项处理 | 明确追问或列出缺失信息 |

### 7.2 评估指标

| 指标 | 说明 |
| --- | --- |
| 意图识别准确率 | 是否正确识别行程规划、知识库、Plan B 等任务 |
| 资料引用准确率 | 引用来源是否真实存在且与结论相关 |
| 记忆提取准确率 | 是否只提取长期有效信息 |
| 推荐可解释性 | 推荐是否说明原因和适配用户约束 |
| 不确定项提示率 | 缺少实时或关键信息时是否明确提示 |
| 高风险操作拦截率 | 是否避免自动执行付款、改签、退票等操作 |
| 幻觉率 | 是否编造价格、天气、政策、开放状态 |
| 结果结构合规率 | 是否按前端卡片需要输出结构化字段 |

### 7.3 验证方法

后续真实接入模型后，建议按以下方式验证：

```text
准备 30-50 条标准测试输入
  ↓
运行 Prompt 得到模型输出
  ↓
按评估指标人工打分
  ↓
记录失败案例
  ↓
迭代 Prompt 和工具约束
  ↓
进入小范围用户灰度测试
```

如果时间允许，可以先用 5-8 条核心测试输入做人工初测，验证 Prompt 是否能稳定输出：

- 正确意图
- 正确引用
- 正确记忆
- 正确不确定项
- 正确安全边界

## 8. 与当前 Demo 的关系

当前 Demo 已落地以下 Prompt 输出结构：

| Demo 位置 | 对应 Prompt |
| --- | --- |
| 发送“新疆 15 天自驾和云南 15 天游学怎么选？” | 意图识别 Prompt、行程规划 Prompt、可信校验 Prompt |

| 执行前确认清单 | Plan B Prompt 和安全边界规则 |

当前 Demo 使用固定样例模拟上述输出。后续真实接入 AI 时，前端卡片结构可以保持不变，只需要将固定样例替换为模型返回的结构化 JSON。
